5. FOLDER STRUCTURE
==================================================

Preserve these main folders:
- `debug`
- `tests`
- `maps`
- `runtime`

But change runtime organization as follows:

Instead of having all runtime classes in one flat runtime folder, split runtime into:
- `runtime/engine`
- `runtime/logic`

Definitions:

A. `runtime/engine`
This folder must contain all classes/scripts that depend directly on Godot engine concepts or engine objects, such as:
- nodes
- scene tree
- `CharacterBody3D`
- `Node3D`
- `MeshInstance3D`
- input APIs
- physics APIs
- raycasts
- camera handling
- scene instancing
- rendering-facing objects
- Godot resources
- environment/debug display hooks tied to the engine
- any code that directly depends on Godot-specific objects or engine APIs

B. `runtime/logic`
This folder must contain pure logic or logic-oriented helpers that do not need to know about Godot engine objects directly, such as:
- calculations
- rules
- pure decision logic
- data transformation
- state transition logic
- reusable logic that could conceptually be written in any language
- code that should not depend on a concrete Godot scene object when avoidable

Important:
- Keep engine-specific code out of `runtime/logic`.
- Move pure logic out of engine-facing classes whenever it is practical.
- Do not force absurd abstraction, but do aggressively extract pure logic when it improves clarity and modularity.
