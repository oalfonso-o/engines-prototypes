## Objetivo de esta iteración

Esta iteración añade soporte visual de wedges, pero NO incluye:

* colisión de wedges
* culling estructural avanzado entre wedges y vecinos
* culling wedge-cube
* culling wedge-wedge
* stitching entre piezas
* deduplicación/merge de caras de wedges
* rotación runtime de wedges
* inferencia heurística de rampas
* optimización de artifact o runtime
* texturas/UVs especiales para wedges

La meta de esta iteración es solo:

* definir wedges explícitas en el TXT
* preprocess visual de esas wedges
* serialización correcta al artifact
* runtime capaz de dibujar `quads + triangles`
* validación matemática de normals y winding

## Regla de scope

Cada wedge debe emitir siempre toda su geometría de plantilla:

* 3 quads
* 2 triangles

En esta iteración no se elimina ninguna cara de wedge por vecindad con cubes ni con otras wedges.

El único culling activo en runtime es el backface culling del renderer.

La meta de esta iteración es validar la definición geométrica y el winding de las wedges, no optimizar su integración estructural con el mapa.

## Regla de culling estructural en esta iteración

En `v11`, el culling estructural del preprocess queda congelado así:

* los cubos siguen usando su regla binaria de visibilidad estructural
* pero esa regla solo considera vecinos tipo cubo sólido
* una celda wedge NO cuenta como vecina sólida para ocultar caras de cubo
* una wedge NO elimina caras de otra wedge
* una wedge NO elimina caras de un cubo
* un cubo NO elimina caras de una wedge

Consecuencia operativa:

* el culling estructural offline de esta iteración solo se aplica a cubo contra cubo
* toda wedge emite siempre su plantilla visual completa:

  * 3 quads
  * 2 triangles

El único culling que afecta a wedges en `v11` es el backface culling del renderer.

Cada celda puede contener:

* vacío

La wedge ocupa exactamente una celda voxel, igual que el cubo.

## Mapping de chars en el TXT

Congelado así:

* `0` = vacío

No hay orientación adicional separada.

### settings.yaml en la raíz del proyecto

Cada proyecto debe tener un `settings.yaml` en su raíz.

* `settings.yaml`
* key raíz: `contract`

La key `contract` debe usar child keys explícitas.

Para `v11`, el bloque mínimo esperado es:

```yaml
contract:
  version: 1
  front_face: ccw
  backface_culling: true
  quad_rule: cross_uv_outward
  triangle_rule: cross_edges_outward
  supports_quads: true
  supports_triangles: true
  runtime_reinterprets_geometry: false
  cube_structural_culling: cube_vs_cube_only
  wedge_structural_culling: none
  wedge_collision: none
  debug_artifact: runtime_world.debug.json

  debug: true
```

Reglas:

* esos valores forman parte del contrato del proyecto
* el preprocess y el runtime deben seguir ese contrato
* el artifact runtime puede repetir una referencia mínima al contrato si se quiere, pero la source of truth formal está en `settings.yaml`
* `preprocessing.debug` controla si el preprocess genera o no el artifact debug paralelo
* en esta spec, el valor inicial esperado es `true`

## Regla de colisión en esta iteración

`v11` no añade colisión de wedges.

Reglas operativas:

* las wedges son piezas visuales solamente
* las wedges NO generan entradas en `boxes`
* las wedges NO modifican las `boxes` existentes del mapa
* las `boxes` del artifact siguen representando solo la colisión de cubos / voxels sólidos no-wedge
* el runtime mantiene exactamente el mismo sistema de colisión que ya existía para cubos

Consecuencia:

* una wedge puede verse en pantalla
* pero en `v11` no aporta colisión propia

### Quads

Los `quads` mantienen exactamente el formato actual de 12 enteros:

`[origin_x, origin_y, origin_z, u_x, u_y, u_z, v_x, v_y, v_z, n_x, n_y, n_z]`

Contrato:

* `p0 = origin`
* `p1 = origin + u_axis`
* `p2 = origin + u_axis + v_axis`
* `p3 = origin + v_axis`
* triángulos:

  * `(p0,p1,p2)`
  * `(p0,p2,p3)`
* `cross(u_axis, v_axis)` apunta en la dirección de `normal`
* visto desde `normal`, el quad es CCW

### Triangles

Los `triangles` usarán 12 enteros con esta forma:

`[p0_x, p0_y, p0_z, p1_x, p1_y, p1_z, p2_x, p2_y, p2_z, n_x, n_y, n_z]`

`triangles` debe tener la misma estructura de agrupación que `quads`:

* `triangles` es un objeto / diccionario
* cada clave es un `material_key`
* cada valor es una lista de triángulos de ese material

Ejemplo conceptual:

* `triangles["gray"] = [triangle0, triangle1, ...]`
* `triangles["green"] = [triangle0, triangle1, ...]`

Reglas de serialización:

* solo aparecen en `triangles` los materiales que tengan al menos un triángulo
* si un material no tiene triángulos, no hace falta incluir esa clave
* `materials` sigue siendo la lista ordenada de materiales usados por el artifact
* el runtime debe iterar `materials` y para cada material consumir:

  * `quads[material_key]` si existe
  * `triangles[material_key]` si existe

Contrato:

* el triángulo es exactamente `(p0,p1,p2)`
* `cross(p1 - p0, p2 - p0)` apunta en la dirección de `normal`
* visto desde `normal`, el triángulo es CCW

## Regla operativa de runtime para wedges

El runtime no reinterpreta wedges.

Reglas:

* los `quads` de wedges siguen exactamente el mismo path de reconstrucción que los `quads` de cubo
* los `triangles` de wedges usan un path específico de reconstrucción de triángulos
* el runtime no rota wedges
* el runtime no recalcula orientación semántica de wedges
* el runtime no recompone wedges a partir de una pieza abstracta
* el runtime no recalcula normales para decidir la cara exterior
* el runtime consume directamente los `quads` y `triangles` ya resueltos por el preprocess

Consecuencia:

* toda la semántica geométrica de la wedge queda congelada en el preprocess y en el artifact
* el runtime solo convierte esa geometría ya decidida en buffers/draw

### Materiales

Las wedges y cubos siguen agrupándose por material igual que el sistema actual.
No estamos cambiando la estrategia de agrupación por material en esta iteración.

### Normales

Las normales no necesitan estar normalizadas a longitud 1.
Basta con que apunten en la dirección correcta y sean coherentes con el winding.

En esta spec, cuando una cara define una normal, esa normal queda fijada de forma exacta.
No se deja como “proporcional a”.

Normales inclinadas congeladas para las wedges:

* `WEDGE_PZ` cara inclinada: `(0,1,-1)`
* `WEDGE_PX` cara inclinada: `(-1,1,0)`
* `WEDGE_NZ` cara inclinada: `(0,1,1)`
* `WEDGE_NX` cara inclinada: `(1,1,0)`

### Coordenadas

Las coordenadas del artifact siguen usando el mismo sistema actual:

* coordenadas en grid / mundo voxel
* no en espacio float final
* el runtime sigue multiplicando por `voxel_size` como hace ahora

## Regla formal de serialización de la wedge

Cada wedge se serializa visualmente exactamente como:

* 3 quads
* 2 triangles

Para evitar reconstrucción implícita en el preprocess, los 3 quads de cada wedge deben quedar definidos también en formato artifact-native:

* `origin`
* `u_axis`
* `v_axis`
* `normal`

Es decir:

* la spec no solo define los triángulos que validan el winding
* la spec también fija directamente cómo debe serializarse cada quad en el artifact

Consecuencia operativa:

* el preprocess no deduce `origin`, `u_axis` o `v_axis` a partir de los triángulos
* el preprocess hace lookup directo de la plantilla de la wedge y emite exactamente esos quads y esos triangles

No se serializa como:

* voxels escalonados
* pieza abstracta pendiente de rotación
* geometría a reinterpretar en runtime

La geometría de la wedge sale ya resuelta desde el preprocess.
